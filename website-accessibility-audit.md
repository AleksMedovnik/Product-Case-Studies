# Расследование недоступности интернет-магазина

**Дагестан, Россия • 2026г.**

| Параметр | Значение |
|----------|----------|
| **География** | Дагестан, Россия |
| **Период проблемы** | Нарастающая недоступность, ~3–4 недели |
| **Мероприятия** | Проведен аудит доступности сайта |
| **Решение** | Подключение DDoS-Guard (российский CDN) |
| **Результат** | Сайт стал доступен для всех пользователей |

---

## 1. Описание проблемы

Интернет-магазин, работающий исключительно на аудиторию Дагестана, начал получать жалобы от пользователей на недоступность сайта. Проблема нарастала постепенно и не была связана с каким-либо конкретным изменением в инфраструктуре.

### Симптомы

- Сайт не открывается вовсе: длительная загрузка или чёрный экран без сообщения об ошибке
- Проблема наблюдалась у части пользователей — не у всех одновременно
- Нет привязки к конкретному оператору связи (МТС, Билайн, Мегафон, WiFi)
- Нет привязки к устройству или браузеру
- На ПК сайт также периодически грузил медленно
- Мобильное приложение для сотрудников работало без сбоев
- VPN иногда помогал открыть сайт
- Два iPhone на одной сети WiFi вели себя по-разному: на одном сайт открывался, на другом нет

---

## 2. Ход расследования

### Этап 1. Сбор данных и первичная диагностика

На начальном этапе был проведён опрос заказчика по ключевым параметрам: характер и масштаб проблемы, изменения в инфраструктуре, симптомы у пользователей, характеристики хостинга.

- Подтверждено: сервер в России (FirstVDS), CDN отсутствует, SSL-сертификат обновлялся, но проблема была и до этого
- DNS-записи не менялись, изменений в коде и конфигурации не проводилось
- Проблема постепенно нарастает — исключает разовый сбой

---

### Этап 2. Проверка доступности сервера

Базовая сетевая диагностика показала, что сервер технически работоспособен:

- `ping` успешен
- `curl -v` — сайт открывается, возвращает HTTP 200
- Трассировка выявила: latency ~204ms до сервера в России (норма 10–30ms) и флаг `!X` на последнем хопе

Флаг `!X` (ICMP Administratively Prohibited) указывал на возможную фильтрацию на уровне firewall или сети провайдера, однако напрямую не влияет на работу HTTPS (порт 443).

---

### Этап 3. Проверка блокировок РКН

Были использованы несколько независимых сервисов проверки блокировок:

| Сервис | Результат |
|--------|-----------|
| `blockinrussia.com` (домен) | Домен может быть заблокирован в РФ |
| `blockinrussia.com` (IP) | Не заблокирован |
| `rkn-test.ru` (домен) | Все параметры в норме, DPI чист |
| `check-host.net` (TCP :443) | Connected из всех точек включая Москву, СПб, Екатеринбург |
| `check-host.net` (HTTP) | 200 OK из всех доступных точек |
| `eais.rkn.gov.ru` | Недоступен |
| `reestr.rublacklist.net` | Недоступен |

**Вывод:** формальной блокировки РКН нет. Сервер глобально доступен. Проблема носит локальный характер.

---

### Этап 4. Проверка гипотезы об устройстве пользователя

Один из пользователей с проблемой использовал iOS 26 Beta. Была выдвинута гипотеза о сетевых багах бета-версии и iCloud Private Relay. Однако:

- Отключение Private Relay не помогло
- Проблема воспроизводилась и на ПК
- Проблема была только с сайтом данного интернет-магазина — другие сайты открывались нормально

**Гипотеза об iOS Beta отклонена.**

---

### Этап 5. Анализ Waterfall — внешние ресурсы

Через WebPageTest (Амстердам) был получен полный Waterfall загрузки страницы (96 запросов, 5MB). Выявлены внешние домены:

- `cdn.jsdelivr.net` — choices.js, загрузка 30–45ms, норма
- `api-maps.yandex.ru` + `yastatic.net` — Яндекс.Карты, самый медленный ресурс: 686ms

Яндекс.Карты стали главным подозреваемым. Однако проверка `check-host.net` по `api-maps.yandex.ru` показала доступность из всех точек. Гипотеза не подтверждена инструментально, но не исключена для конкретных операторов Дагестана.

---

### Этап 6. Диагностика с живым пользователем

Пользователь с проблемой выполнил проверку через `check-host.net`. Результат: TCP Connected и HTTP 200 из всех точек, включая российские. Это означало, что стандартные инструменты мониторинга не воспроизводят проблему из-за отсутствия точек в Дагестане.

---

## 3. Установленная причина

Совокупность признаков указала на локальную фильтрацию трафика на уровне **ТСПУ** (Технических Средств Противодействия Угрозам) у дагестанских операторов связи:

| Признак | Интерпретация |
|---------|---------------|
| VPN помогал открыть сайт | Трафик обходит локальный фильтр через VPN |
| Два iPhone на одном WiFi — разный результат | ТСПУ фильтрует избирательно, возможно по TLS fingerprint устройства |
| Нет привязки к оператору или браузеру | Фильтрация на уровне сети, а не приложения |
| Проблема нарастает постепенно | Расширение зон фильтрации или ужесточение правил ТСПУ |
| `check-host.net` не воспроизводит | Нет точек мониторинга в сети дагестанских операторов |
| Сервер технически доступен | Проблема не на стороне хостинга |

**Точная причина** попадания под фильтрацию ТСПУ не установлена — это может быть косвенное попадание IP в расширенные списки, совпадение с паттернами фильтруемого трафика или локальные правила конкретных операторов.
